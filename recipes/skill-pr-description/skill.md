# PR Description

Produce a polished pull request description by reading the current branch's diff against `main` (or `master`).

## Steps

1. Confirm the current directory is a git repo and that there are commits ahead of base (`git rev-list --count <base>..HEAD`). If zero, abort and tell the user.
2. Gather context in parallel:
   - `git log <base>..HEAD --pretty=format:'%h %s%n%b' --no-merges`
   - `git diff <base>...HEAD --stat`
   - `git diff <base>...HEAD --name-status`
   - `git diff <base>...HEAD` (full diff — sample only; do not load the entire payload into the response)
3. Infer the change type from commit subjects:
   - `feat` / `add` → new feature
   - `fix` → bug fix
   - `refactor` / `chore` → maintenance
   - mixed → "Multiple changes"
4. Write the PR body using this template:

```markdown
## Summary

<2–4 bullets describing what changed and why, in plain language>

## Changes

- <high-level area 1: brief>
- <high-level area 2: brief>

## Test plan

- [ ] <unit / integration test additions>
- [ ] <manual smoke test, if any>
- [ ] <CI checks expected to pass>

## Breaking changes

<None — or list and describe migration path>

## Screenshots / recordings

<Insert if UI changes; otherwise: N/A>
```

5. Output the body verbatim in a fenced code block so the user can copy-paste into GitHub.
6. Suggest a PR title under 70 characters using a Conventional Commit prefix matching the dominant change type.

## Rules

- Do NOT invent test plans — only list checks the user can plausibly run based on what changed.
- Do NOT mention "Generated by Claude" or attribution lines.
- If the diff includes new dependencies (`package.json`, `requirements.txt`, etc.), call them out explicitly under **Changes**.
- If the diff touches migration files (`migrations/`, `*.sql`), include a "Migration safety" callout under **Breaking changes**.
- Keep the entire body under ~80 lines.
